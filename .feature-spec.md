# Ping/Pong Mechanism for Inactive Client Disconnection

GitHub Issue: https://github.com/joebew42/ex_chat/issues/13

## Overview

Introduce a server-initiated ping/pong mechanism between client and server to detect and disconnect inactive clients, as described in the [Cowboy WebSocket docs](https://ninenines.eu/docs/en/cowboy/2.4/guide/ws_handlers/#_keeping_the_connection_alive).

## Current Behavior

- The WebSocket handler sets `idle_timeout: 10 minutes`.
- If no frames arrive in that window, Cowboy closes the connection.
- Neither the server nor the client actively checks liveness.

## Desired Behavior

- The server periodically sends WebSocket ping frames to connected clients.
- Browsers respond to ping frames with pong automatically (protocol-level).
- Each pong resets Cowboy's `idle_timeout`, keeping the connection alive.
- Clients that fail to respond to pings get disconnected after the idle timeout.

## Plan

### 1. Server-side: `lib/ex_chat/web/websocket_controller.ex`

- On `websocket_init/1`, start a recurring timer using `Process.send_after(self(), :ping, @ping_interval)`.
- Add a `websocket_info(:ping, state)` clause that:
  - Sends a WebSocket `:ping` frame to the client.
  - Re-schedules the next `:ping` timer.
  - Returns `{:reply, {:ping, ""}, state}`.
- Reduce the `idle_timeout` from 10 minutes to 60 seconds.
- Extract timeout values as module attributes (`@ping_interval`, `@idle_timeout`).

### 2. Client-side: `priv/static/chat.html`

- No changes needed. Browsers handle WebSocket ping/pong at the protocol level automatically.

### 3. Tests: `test/ex_chat/web/websocket_acceptance_test.exs`

- Test that a connected client receives ping frames (or stays connected beyond the idle timeout thanks to pings).
- Test that the connection is closed when the client is truly inactive (doesn't respond to pings).
